¿Qué es RSA?

Rivest-Shamir-Adleman

RSA es el primer algoritmo de cifrado asimétrico practicable y es el más utilizado en todo el mundo. Desarrollado en 1977 por Ron Rivest, Adi Shamir y Leonard Adleman en el MIT, RSA se basa en la dificultad matemática de factorizar números enteros grandes y es fundamental para la seguridad de internet.

Características principales

  • Algoritmo asimétrico: Clave pública/privada diferente
  • Basado en factorización: Dificultad de factorizar números grandes
  • Longitud de clave variable: 1024, 2048, 3072, 4096 bits
  • Usos múltiples: Cifrado, firma digital, intercambio de claves
  • Patente expirada: Libre de patentes desde 2000
  • Estándar PKCS #1: RFC 8017 (actualización de PKCS#1 v2.2)

Cómo funciona RSA

Fundamentos matemáticos

RSA se basa en la teoría de números y la aritmética modular. Su seguridad depende de la dificultad de factorizar el producto de dos números primos grandes.

Generación de claves

  1. Seleccionar primos: Elegir dos números primos grandes p y q
  2. Calcular módulo: n = p × q (clave pública)
  3. Función totiente: φ(n) = (p-1) × (q-1)
  4. Exponente público: e (normalmente 65537)
  5. Exponente privado: d = e^(-1) mod φ(n)

Proceso de cifrado

  • Mensaje original: M (número menor que n)
  • Cifrado: C = M^e mod n
  • Descifrado: M = C^d mod n

Firma digital

  • Firma: S = H(M)^d mod n (usando clave privada)
  • Verificación: H(M) = S^e mod n (usando clave pública)

Uso en Copias de Seguridad

Aplicaciones principales

  • Intercambio de claves: Establecer claves simétricas seguras
  • Firma digital: Verificar integridad e autenticidad
  • Autenticación: Probar identidad sin revelar clave privada
  • Certificados SSL/TLS: Base de HTTPS y comunicaciones seguras
  • Backup en nube: Cifrado de claves de acceso
  • Firma de software: Verificar autenticidad de actualizaciones

Ejemplo de uso

# Generar claves RSA
openssl genpkey -algorithm RSA -out private_key.pem -pkcs8 -outform PEM

# Extraer clave pública
openssl rsa -pubout -in private_key.pem -out public_key.pem

# Cifrar archivo con clave pública
openssl rsautl -encrypt -pubin -inkey public_key.pem -in archivo.txt -out archivo.enc

# Descifrar con clave privada
openssl rsautl -decrypt -inkey private_key.pem -in archivo.enc -out archivo.txt

# Python con cryptography
from cryptography.hazmat.primitives.asymmetric import rsa, padding
from cryptography.hazmat.primitives import hashes

# Generar claves
private_key = rsa.generate_private_key(
    public_exponent=65537,
    key_size=2048
)
public_key = private_key.public_key()

# Cifrar
ciphertext = public_key.encrypt(
    message,
    padding.OAEP(
        mgf=padding.MGF1(algorithm=hashes.SHA256()),
        algorithm=hashes.SHA256(),
        label=None
    )
)

Seguridad y Rendimiento

Nivel de seguridad

  • Seguridad probada: 40+ años de análisis criptográfico
  • Ampliamente estudiado: Millones de horas de análisis
  • Resistente a ataques actuales: No vulnerable a ataques conocidos
  • Mejora continua: Aumento de tamaño de clave recomendado
  • Híbrido recomendado: Usar con algoritmos simétricos

Consideraciones de rendimiento

  • Lento para datos grandes: Recomendado solo para datos pequeños
  • Cálculo intensivo: Operaciones exponenciales costosas
  • Mejora con hardware: Aceleración en HSM dedicados
  • Recomendación actual: Mínimo 2048 bits, preferible 3072+

Tamaños de Clave RSA

Recomendaciones por período

  • 1024 bits: Obsoleto, no usar (quebrado teóricamente)
  • 2048 bits: Mínimo actual para uso general
  • 3072 bits: Recomendado para datos sensibles
  • 4096 bits: Máxima seguridad, uso limitado

Recomendaciones por sector

  • SSL/TLS: 2048 bits (mínimo aceptable)
  • Gobierno: 3072 bits o superior
  • Banca: 3072 bits para transacciones
  • Militar: 4096 bits para información clasificada

Transición recomendada

  • Planificar migración de 1024-bit antes de 2015
  • Actualizar certificados SSL regularmente
  • Monitorear avances en factorización
  • Considerar post-quantum cryptography para futuro

Implementación Práctica

Librerías y frameworks

  • OpenSSL: Implementación de referencia más utilizada
  • Python cryptography: Librería moderna y segura
  • Java Cryptography Extension: API estándar de Java
  • .NET System.Security: Clases de cifrado nativas
  • Bouncy Castle: Implementación completa multiplataforma
  • LibreSSL: Versión fork de OpenSSL
  • GnuTLS: Alternativa open source

Modos de operación

  • PKCS #1 v1.5: Modo tradicional (legacy)
  • OAEP: Optimal Asymmetric Encryption Padding (recomendado)
  • PKCS #1 v2.1: Estándar actual RFC 8017
  • SSL/TLS: Integración en protocolos de red

Herramientas que usan RSA

Protocolos y estándares

  • HTTPS: Base de certificados SSL/TLS
  • SSH: Autenticación y cifrado de conexiones
  • PGP/GPG: Cifrado de email y archivos
  • S/MIME: Firma y cifrado de email
  • IPSec: VPN y comunicaciones de red seguras

Software de seguridad

  • Apache/Nginx: Servidores web con soporte SSL/TLS

Gestión de certificados

Cumplimiento y Estándares

Estándares que lo reconocen

  • RFC 8017: PKCS #1 v2.2, especificación oficial
  • FIPS 186-4: Estándar federal de procesamiento de información
  • ISO/IEC 18033-2: Estándar internacional de cifrado asimétrico
  • NIST SP 800-57: Guía de gestión de claves
  • PCI DSS: Requerido para procesamiento de tarjetas

Requisitos por sector

  • Finanzas: 2048 bits mínimo, 3072 recomendado
  • Gobierno: 3072 bits para datos sensibles
  • Salud: Cumple HIPAA para datos protegidos
  • eCommerce: Obligatorio para transacciones PCI DSS
  • Cloud: Estándar para autenticación de usuarios

Desafíos y Limitaciones

Problemas conocidos

  • Rendimiento lento: No adecuado para cifrar datos grandes
  • Vulnerable a ataques de temporización: Necesita contramedidas
  • Problemas con padding: PKCS #1 v1.5 tiene vulnerabilidades
  • Quantum computing: Vulnerable a algoritmos de Shor
  • Gestión de claves compleja: Claves grandes y costosas de generar

Soluciones y mejores prácticas

  • Usar OAEP padding: Para cifrado de datos
  • Implementar blinding: Proteger contra ataques de temporización
  • Combinar con simétricos: RSA para intercambio de claves, AES para datos
  • Rotación regular: Cambiar claves periódicamente
  • Hardware security modules: Aceleración y protección de claves

Mejores Prácticas

Implementación segura

  • Usar claves de 2048 bits mínimo: 3072+ para datos sensibles
  • Implementar OAEP padding: Evitar PKCS #1 v1.5
  • Proteger contra timing attacks: Usar blinding o HSM
  • Validar implementaciones: Verificar contra vulnerabilidades conocidas
  • Monitorear avances: Mantenerse actualizado con factorización

Consideraciones operativas

  • Generación de claves: Usar generadores seguros de números aleatorios
  • Almacenamiento seguro: Proteger claves privadas con HSM
  • Backup de claves: Almacenamiento seguro y recuperación de desastres
  • Auditoría: Registrar uso de claves y operaciones
  • Certificados: Gestionar lifecycle completo de certificados

Futuro y Consideraciones

Evolución del algoritmo

  • Migración a ECC: Curvas elípticas más eficientes
  • Post-quantum cryptography: Preparación para computación cuántica
  • Mejores tamaños de clave: 3072 bits como nuevo estándar
  • Hardware acceleration: Mejor soporte en procesadores futuros

Mejores prácticas actuales

  • Usar RSA solo para firma digital y intercambio de claves
  • Implementar criptografía híbrida para cifrado de datos
  • Planificar migración a algoritmos post-quantum
  • Monitorear avances en factorización de números grandes
  • Evaluar ECDSA/Ed25519 como alternativas más eficientes
